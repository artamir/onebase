## 🔴 Критические замечания

### 1. Path Traversal в file_builtins.go (БЕЗОПАСНОСТЬ) — ✅ ИСПРАВЛЕНО

> Решено в `b20f388`: общий `resolveSafePath` + `SetFileSandbox(корень)`; маршрутизированы ВСЕ файловые builtins (включая `builtins_files.go`); sandbox включается в demo-режиме (`cli.runServer`). По умолчанию выключен для доверенного desktop.

**Файл**: `internal/dsl/interpreter/file_builtins.go:56, 135`

ЧтениеТекста/ЗаписьТекста не валидируют путь — DSL-код может читать/писать произвольные файлы на сервере:
```onec
ф = Новый ЧтениеТекста("/etc/passwd")
// или
ф = Новый ЧтениеТекста("C:\Windows\System32\config\SAM")
```

**Предложение**: Добавить whitelist директории (например, только внутри data-dir базы). Проверять через `filepath.Rel()` + убедиться что результат не содержит `..`.

### 2. Nil-паники и silent 0 в math builtins — ⏸ ОТКЛОНЕНО как P0

> Завышенный приоритет. Весь слой builtins намеренно «мягкий» (`strArg`→`""`, `floatArg`→`0`); строгая валидация только Log/Sqrt непоследовательна. Фактическая неточность ревью: `Log()` без аргумента даёт `-Inf`, а не `0`. Решать единой политикой валидации аргументов или не трогать.

**Файл**: `internal/dsl/interpreter/builtins_stage_a.go`

Функция `strArg(args, 0)` безопасна (возвращает `""` при nil), НО:
- `floatArg(args, 1)` в `strGetLineFn`/`charCodeFn` при недостатке аргументов вернёт `0`, что молча даёт неверный результат вместо понятной ошибки
- `mathUn`/`mathBin` молча возвращают 0 при nil-аргументах — `Sqrt(nil)` = 0 вместо ошибки. Для `Log` и `Sqrt` это математически некорректно

**Предложение**: Добавить проверки `len(args) < N` с возвратом ошибки для ключевых функций. Для Log/Sqrt — возвращать ошибку при ≤ 0.

### 3. Fill без транзакции — race condition

**Файл**: `internal/entityservice/service.go:287–303`

`GetByID` и `GetTablePartRows` в `Fill()` выполняются вне транзакции. Между загрузкой источника и вызовом хука данные могут измениться. Для ввода на основании это допустимо (мы только читаем), но стоит добавить комментарий о принятом решении.
