## 2. Статья для Инфостарт

### OneBase за десять дней: управляемые формы, 14 языков интерфейса, точные деньги и инструменты для разработки с ИИ

**Аннотация.** OneBase — открытая бизнес-платформа на Go со знакомым учётным разработчикам DSL: справочники, документы, регистры, отчёты, проведение, запросы. За десять дней после релиза v0.3.27 (22–31 мая) в проект влилось больше сотни изменений и почти 39 тысяч строк кода. Рассказываю, что появилось: от полноценных управляемых форм с двусторонним конвертером и точной денежной арифметики на decimal до CLI-инструментов, благодаря которым конфигурацию для платформы может писать связка «разработчик + ИИ».

#### Зачем это всё

OneBase я развиваю в свободное время как ответ на простой вопрос: можно ли взять привычную учётному разработчику модель (платформа → конфигурация → информационная база, DSL «как у всех») и реализовать её на современном открытом стеке — Go + PostgreSQL, веб-интерфейс, всё под git и MIT-лицензией. Прошлая статья на Инфостарт описывала фундамент. Эта — про то, как за одну насыщенную неделю платформа догнала по эргономике ещё несколько «взрослых» возможностей.

Пройдусь по главному.

#### 1. Управляемые формы — теперь по-настоящему

Раньше формы в OneBase были упрощёнными: массив видимых полей в YAML и автогенерируемый рендер. Этого хватало для CRUD, но не для реальных прикладных форм с вкладками, командами и обработчиками событий.

За неделю появился полноценный объект метаданных **«управляемая форма»**:

- **декларативное описание** структуры формы в YAML рядом с DSL-модулем (`.form.os`);
- **UI-редактор в конфигураторе** по тому же паттерну, что и редактор печатных форм — split-pane: слева код/структура в Monaco, справа живой предпросмотр;
- **рантайм событий** — `ПриОткрытииФормы`, обработчики команд, объект-обёртка `Объект`, работа с табличными частями (`ТЧ.Добавить`), движения регистров прямо из формы, корректные ISO-даты;
- **двусторонний конвертер** управляемых форм: импорт из XML-формата Enterprise-системы и экспорт обратно. Конвертер построен на нейтральном промежуточном представлении (IR): `Form.xml ↔ IR ↔ .form.yaml`, с сохранением порядка элементов, локализованных заголовков и round-trip-идентичности.

Параллельно управляемые формы получили и **обработки** (DataProcessors, План 40): у обработок появились табличные части, типы параметров (`text`/`choice`/`reference`/`file`) и собственные формы.

Юридически конвертер опирается только на открытые текстовые форматы выгрузки (XML и BSL) и собственные таблицы соответствий — без чужих XSD и без реверс-инжиниринга исполнения.

#### 2. Интерфейс на 14 языках и многоязычные синонимы метаданных

Большой блок (PR #10) — интернационализация. Появилась локализация UI на немецкий, сербский и ещё 12 языков: грузинский, армянский, испанский, казахский, азербайджанский, узбекский, украинский, турецкий, румынский, португальский, китайский, французский.

Но интереснее другое: **синонимы метаданных теперь многоязычные**. Наименование справочника, реквизита, документа, регистра, отчёта, подсистемы можно задать на нескольких языках в блоке `titles:`, и рантайм подставит нужный по языку пользователя (с откатом на дефолтный `title:`, если перевода нет). Пока переводы синонимов вписываются в YAML руками — UI-редактор для них в конфигураторе ещё в планах (План 31, статус «Не начато»).

Чтобы переводы интерфейса не разъезжались, я добавил инструмент `i18ncheck` и повесил его в pre-commit hook — недостающие или лишние ключи в локалях ловятся ещё до коммита.

#### 3. Роли и права (RBAC)

Управление ролями переехало в конфигуратор, и — что важнее — права начали **реально ограничивать интерфейс** (PR #9), а не просто храниться в метаданных. Это первый подход к полноценной ролевой модели.

#### 4. Инструменты для разработки с ИИ

Отдельная история, которой я особенно доволен. Узкое место при работе ИИ-ассистента над конфигурацией — не «умение кодить», а **петля обратной связи**: без headless-способа увидеть структуру и быстро поймать ошибку ИИ работает вслепую.

Итерация 1 (План 41), принцип «CLI-first — команда, текст/JSON, точная ошибка»:

- **`onebase check`** — гейт валидации: синтаксис `.os`, неизвестные функции, YAML-схема всех видов объектов, компиляция запросов — всё с `file:line:col` и ненулевым exit-кодом. Готовый pre-commit/CI-гейт.
- **`onebase describe --json`** — «рентген» конфигурации: стабильный read-only JSON со всеми сущностями, процедурами модулей и списком встроенных функций.
- **`onebase ai-guide`** — справочник по платформе, генерируемый из неё самой (список builtins группируется по источникам автоматически и не устаревает).
- **`onebase init`** теперь кладёт в новый проект `AGENTS.md` — ИИ-ассистент сразу ориентируется в конвенциях репозитория.

Плюс к этому — команда **`onebase procrun`** для офлайн-отладки обработок без поднятия всей базы.

#### 5. DSL стал богаче

Язык дорос ещё на несколько этапов дорожной карты (План 36):

- недостающие строковые, датовые и математические функции;
- `ИнформацияОбОшибке`, поиск по реквизиту, методы объектов;
- функции `Символ`/`Chr`/`Char`, чтение/запись текстовых файлов;
- **модуль менеджера** (`X.manager.os`) — вызовы вида `Документы.X.Метод()`;
- `Ссылка.ПолучитьОбъект()` возвращает изменяемый объект (issue #8) — со всей цепочкой исправлений, чтобы ссылки «несли менеджера» и обогащались данными шапки и табличных частей;
- тип **`ТаблицаЗначений`** (этап D) с полным набором методов: колонки, добавление строк, `Итог`, `Найти`/`НайтиСтроки`, `Сортировать`, `Свернуть`, выгрузка/загрузка колонок, итерация `Для Каждого`;
- чтение **регистров накопления** прямо из DSL (этап E): `РегистрыНакопления.X.Остатки()`, `.Движения()`, `.ВыбратьПоРегистратору()`;
- мелкие добивки: `Массив.Найти/Установить/ВГраница/Вставить`, `НайтиФайлы`, `Вычислить`.

Поправил и пару коварных мелочей: `Сред` без третьего аргумента, присваивание поля структуры через точку, а nil-операнд в арифметике теперь честно трактуется как 0, а не превращает выражение в конкатенацию строк.

И главное для учётной системы — **деньги без float**. Числа DSL переведены с `float64` на `shopspring/decimal`: литералы, арифметика, сравнения, builtins (`Число`, `Окр` с режимами, `Абс`, `Цел`, `Макс`, `Мин`). decimal живёт сквозь всю границу БД: PostgreSQL `NUMERIC` ↔ decimal без промежуточного float, SQLite `TEXT` ↔ decimal по типу поля. Это закрывает классический баг `0.1 * 1.05`, дающий `0.10500000000000001`. Заодно `ЧислоПрописью` пересчитывает копейки точным decimal-умножением. На уровне метаданных к этому добавилась **разрядность**: тип `number(10,2)` (или `decimal(15,2)`) задаёт длину и точность, конфигуратор показывает поля «Длина»/«Точность», DDL для PostgreSQL генерирует `NUMERIC(L,P)`, а при записи значение округляется до `Scale` и проверяется на переполнение по `Length`.

#### 6. REST API v2 и единый сервис записи

Я вынес общую логику записи в `EntityService` — единую точку для упсёрта, табличных частей, движений и проведения. Теперь REST API использует тот же сервис, что и UI: одинаковое поведение `OnWrite`, проведение, версионирование. Заодно закрыл неприятный баг — потерю табличных частей при POST/UPDATE через REST и не-сброс регистров при отмене проведения. Покрыл новое поведение интеграционными тестами.

#### 7. SQLite, конкурентность и кодировки

- `SQLITE_BUSY` под конкурентной нагрузкой побеждён (+ concurrency-тест);
- путь к файлу SQLite нормализуется при добавлении базы;
- при загрузке файлов автоматически определяется кодировка Windows-1251 — больше не нужно вручную перекодировать выгрузки.

#### 8. Десятки улучшений эргономики

Виджеты главной страницы по подсистемам, inline-создание элемента справочника прямо из ссылочного поля, кнопка-лупа для открытия карточки, перемещение баз вверх/вниз, оптимистические блокировки документов и справочников, настройка размера страницы списков, явные стрелки сворачивания в дереве конфигуратора, иконки объектов в дереве (справочники 📕, документы 📄). Плюс длинный хвост багофиксов: от «кнопки не работают из-за локали O'zbekcha» и неработающей кнопки «Пароль» в WebView2 до диалога выбора папки на Windows Server 2019 и регистронезависимых отборов по кириллице. Отдельно поправил неприятный для автоматизации момент: при headless/CI-запуске любая ошибка команды раньше вылезала модальным окном `MessageBox`, которое некому было закрыть, — теперь окно показывается только при запуске двойным кликом без консоли, а в пайпе/файле ошибка честно идёт в `stderr` (флаг `--no-gui` / `ONEBASE_NO_GUI`).

#### 9. Конструктор рабочего стола

Виджеты главной страницы из прошлого пункта обзавелись нормальным редактором. У рабочего стола теперь два режима раскладки: **«Авто»** — виджеты текут потоком по ширине (поведение по умолчанию), и **«По рядам»** — WYSIWYG-конструктор с перетаскиванием: чипы виджетов таскаются мышью между рядами и пулом неиспользованных, ряды добавляются и удаляются, порядок и группировка сохраняются ровно так, как настроил пользователь (в JSON, без алфавитной пересортировки). Тот же визуальный редактор получили и рабочий стол подсистемы, и глобальная «Главная»; сырой YAML никуда не делся — он спрятан в спойлер «Расширенно». Заодно навигация стала нейтральной: при входе на `/ui/` больше не подсвечивается «первая попавшаяся» подсистема — активна «Главная», а сайдбар плоский.

#### 10. Качество: тесты, hardening и фиксация техдолга

Часть недели ушла не на фичи, а на укрепление фундамента — по итогам внутреннего аудита.

- **Точные деньги (decimal)** — см. раздел про DSL: это, по сути, тоже hardening, только финансовый.
- **HTTP-таймауты** на api- и launcher-серверах (`ReadHeaderTimeout`, `IdleTimeout`) — защита от Slowloris и зависших keep-alive. `ReadTimeout`/`WriteTimeout` я намеренно не выставлял: они оборвали бы загрузку крупных `.obz`, SSE-стрим отладчика и скачивание бэкапов — тело запроса ограничивается отдельными `MaxBytesReader`.
- **Страж глубины рекурсии в DSL** — без него бесконечный самовызов процедуры рос стек горутины до краха процесса (panic вне досягаемости `Попытки`); теперь есть порог `MaxRecursionDepth`.
- **Sandbox файловых путей** для файловых builtins (`ЧтениеТекста`/`ЗаписьТекста`/`НайтиФайлы` и пр.) — защита от path traversal; по умолчанию выключен для доверенного desktop, включается в демо-режиме, где обработки исполняет недоверенный пользователь.
- **Покрытие тестами** критичных путей: многопользовательский `auth` на SQLite (7.9% → 61.3%) и конвертер из 1С (`converter` 0% → 63.5%, `parser1c` 0% → 24.1%).
- **Гигиена кодовой базы**: снят UTF-8 BOM с 41 `.go`-файла, а в CI добавлен страж, который падает при возврате BOM. Остаток техдолга зафиксирован отдельным планом (43): полный graceful shutdown, единый `slog`, раскол монолитов, добор покрытия.

- **4** влитых PR, закрыты issue **#8** и **#13**;

### 4.2. DSL обработки — Запрос, Массив, Структура, Попытка/Исключение
`examples/trade/src/пересчётЦен.proc.os` — модуль обработки, отлаживается офлайн через `onebase procrun`:

```bsl
Процедура Выполнить()
  Процент = Параметры.Процент;
  Если Процент = 0 Тогда
    Error("Процент не может быть нулевым");
  КонецЕсли;

  Запрос = Новый Запрос;
  Запрос.Текст = "ВЫБРАТЬ Наименование, ЦенаПродажи ИЗ Номенклатура";

  Попытка
    Результат = Запрос.Выполнить();
  Исключение
    Сообщить("Ошибка запроса: " + ОписаниеОшибки());
    Возврат;
  КонецПопытки;

  Изменения = Новый Массив;
  Для Каждого Строка Из Результат Цикл
    Если Строка.ЦенаПродажи > 0 Тогда
      Элемент = Новый Структура;
      Элемент.Наименование = Строка.Наименование;
      Элемент.Было = Строка.ЦенаПродажи;
      Элемент.Стало = Строка.ЦенаПродажи * (1 + Процент / 100);
      Изменения.Добавить(Элемент);
    КонецЕсли;
  КонецЦикла;
  // ... вывод результата
КонецПроцедуры
```

Резолвинг заголовка: `Titles[lang]` побеждает, иначе откат на дефолтный `title:` (поведение из `internal/metadata/types_test.go`). В YAML это выглядит так:

### 4.7. ТаблицаЗначений (этап D)
Универсальный контейнер с колонками — для агрегации и промежуточных расчётов в DSL (сквозной сценарий из `internal/dsl/interpreter/valuetable_dsl_test.go`):

```bsl
Т = Новый ТаблицаЗначений;
Т.Колонки.Добавить("Товар");
Т.Колонки.Добавить("Сумма");

Стр = Т.Добавить();

```

Поддерживаются `Найти`/`НайтиСтроки(Отбор)`, `Сортировать("Сумма Убыв")`, `Свернуть("Товар", "Сумма")`, `ВыгрузитьКолонку`/`ЗагрузитьКолонку` и итерация `Для Каждого Строка Из Т Цикл`.

### 4.8. Точные деньги (decimal) и разрядность поля (План 42)
Арифметика DSL работает на `shopspring/decimal`, поэтому классический «грязный» float здесь невозможен (поведение из `internal/dsl/interpreter/amount_in_words_test.go` и тестов builtins):

```bsl
Возврат 0.1 * 1.05;   // → 0.105, а не 0.10500000000000001
```

А на уровне метаданных тип числа несёт длину и точность (`parseNumberSpec` в `internal/metadata/yaml.go`, поведение записи — `internal/storage/crud.go`):

```yaml
fields:
  - name: Цена
    type: number(10,2)        # NUMERIC(10,2) в PostgreSQL
  - name: Курс
    type: decimal(15,4)
```

При записи значение округляется до `Scale` и проверяется на переполнение по `Length` — лишние знаки не «протекают» в базу, а слишком большое число даёт явную ошибку, а не молчаливое искажение.

## 5. Результаты сверки заявлений с кодом

После замечания про «редактор переводов» я перепроверил ключевые фичи по фактическому коду и тестам:

| Заявление в статье | Статус | Чем подтверждается |
|---|---|---|
| AI-инструменты: `check` / `describe` / `ai-guide` / `procrun` | ✅ реализовано | `internal/cli/{check,describe,aiguide,procrun}.go`, все через `rootCmd.AddCommand(...)` |
| RBAC реально ограничивает интерфейс | ✅ реализовано | `Server.can()` → `user.Has(kind,entity,op)`, `requirePerm` отдаёт 403; админ-роуты `/ui/admin/roles`. *Нюанс:* при работе без пользователей (kiosk) доступ открыт — это by design |
| Управляемые формы: рантайм + редактор + конвертер | ✅ реализовано | `handlers_managed_events.go` (`ExecuteEventHandler`), пакет `internal/onec_forms` с `reader_xml`/`writer_xml`, фасад `ImportFromOneC`/`ExportToOneC`/`Validate` |
| Двусторонний конвертер форм (round-trip) | ✅ реализовано | reader+writer для XML и YAML, golden round-trip-тесты в `internal/onec_forms` |
| Многоязычные синонимы метаданных | ✅ реализовано | `Entity.Titles` + резолв по языку (`types_test.go`), хранение в `titles:` YAML |
| Редактор переводов синонимов в конфигураторе | ❌ **не реализовано** | План 31, статус «Не начато»; за неделю только `docs(plans)` |
| ТаблицаЗначений / регистры в DSL / мелкие добивки | ✅ реализовано | `valuetable.go` + тесты, `dsl_registers.go` + тесты, `collections.go`/`builtins_files.go` + тесты |
| Точная decimal-арифметика + `ЧислоПрописью` (План 42) | ✅ реализовано | `shopspring/decimal` в `internal/dsl/interpreter` (`builtins.go`, `amount_in_words.go` + тесты) |
| Числовые поля `number(L,P)` — округление, переполнение, `NUMERIC(L,P)` | ✅ реализовано | `parseNumberSpec`/`applyNumberSpec` (`internal/metadata/yaml.go`, `number_spec_test.go`, `internal/storage/crud.go`) |
| Страж глубины рекурсии DSL (`MaxRecursionDepth`) | ✅ реализовано | `internal/dsl/interpreter/{interpreter,limits}.go`, `recursion_test.go` |
| Sandbox файловых путей для builtins | ✅ реализовано | `filesandbox.go` + `filesandbox_test.go`; включение через `cli.runServer → SetFileSandbox` |
| Конструктор рабочего стола (раскладки `auto`/`rows`, drag) | ✅ реализовано | `internal/metadata/{homepage,subsystem}.go` + тесты, `internal/launcher/{widget_handlers,configurator_tmpl}.go` |

Все пункты, кроме редактора переводов синонимов, подтверждены реализацией и тестами.
