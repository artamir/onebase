## Контекст и цель

Этап 2a даёт бэкенд генерации (`ai-generate` возвращает предложенный diff `changes`).
Этап 2b замыкает фичу end-to-end: **панель генерации в конфигураторе** показывает diff
и даёт **применить** его в рабочую конфигурацию.

**Объём (решение пользователя):** применение — **сначала файловые базы**; БД-базы —
понятная ошибка «пока не поддерживается». Один PR: бэкенд apply + UI.

## Целевая архитектура

### 2b-1 — бэкенд применения (`internal/launcher/ai_apply.go`)
Эндпоинт `POST /bases/{id}/configurator/ai-apply` (та же admin-зона конфигуратора).
Тело: `{ "changes": [ { "path": "catalogs/клиент.yaml", "newContent": "..." } ] }`
(подмножество `GenChange` из 2a — нужны только `path` и `newContent`).

- **Валидация пути** `validApplyPath(rel)` — путь обязан быть
  `<метаданные-подкаталог>/<безопасное-имя>.yaml`, где подкаталог ∈
  {catalogs, documents, registers, inforegs, enums, accounts, accountregs}
  (как в 2a `kindSubdir`), имя проходит `safeFileName` (нет `..`/слешей/пустого).
  Любой другой путь → ошибка (нет записи вне метаданных конфига).
- **БД-базы:** `b.ConfigSource == "database"` → ответ-ошибка «применение в БД-базы пока
  не поддерживается; используйте файловую базу» (best-effort/позже).
- **Запись (файловая база):** `os.WriteFile(filepath.Join(b.Path, FromSlash(rel)), newContent)`
  + защита: итоговый путь обязан лежать внутри `b.Path` (как в 2a с overlay).
  Создаём подкаталог при необходимости.
- Ответ: `{ ok: true, applied: N }` либо `{ error, applied: <успевшие> }`.

После применения объекты появляются в дереве конфигуратора (он читает `b.Path` живьём).
Перезагрузку рабочей базы (hot-reload/restart) НЕ трогаем — существующий механизм, как
при любой правке конфигурации.

### 2b-2 — панель генерации (`internal/launcher/configurator_tmpl.go`)
Отдельная плавающая кнопка `#cfggen-btn` + панель `#cfggen-panel` со своим IIFE
(не трогает существующий `#cfgai-panel` ассистента — изоляция JS). Поток:
1. ввод ТЗ → `POST ai-generate` → ответ `{text, changes}`.
2. рендер diff: список `changes` (путь + «новый/изменён» + содержимое в `<pre>`,
   сворачиваемо). Текст модели — над списком.
3. кнопки **«Применить»** (POST `ai-apply` с `changes`) и **«Отклонить»** (очистка).
4. при успехе — сообщение «применено N» + `location.reload()` (дерево обновляется).

Минимум стилей, экранирование содержимого через `textContent` (как у ассистента — без
XSS). Кнопка генерации видна, только если ИИ настроен (`ai-enabled`, как у ассистента).

## Тесты

- **2b-1 (`internal/launcher/ai_apply_test.go`):** на временной файловой базе (`*Base{Path,
  ConfigSource:"file"}` + `*handler`): apply одного `change` → файл записан в `b.Path/catalogs/…`;
  путь с `..`/чужим подкаталогом → ошибка, файл не создан; `validApplyPath` юнит-тестом на
  наборе плохих/хороших путей. (Если конструировать `*handler`+`*Base` в тесте тяжело —
  тестировать `validApplyPath` и чистую функцию записи `applyChanges(baseDir, changes)`
  напрямую, а хендлер оставить тонкой обёрткой.)
- **2b-2 (структурный, `renderCfgFoot`):** rendered cfg-foot содержит `cfggen-panel`,
  `cfggen-prompt`, `cfggen-send`, fetch на `ai-generate` и `ai-apply`, кнопку «Применить».
- **Ручная приёмка (в браузере, отмечено как ручной шаг):** в конфигураторе открыть панель
  генерации, «нужен справочник Клиенты и документ Заявка с товарами» → виден diff →
  «Применить» → объекты появились в дереве, база стартует.

## Безопасность

- `ai-apply` валидирует каждый путь строго (метаданные-подкаталог + безопасное имя);
  запись только внутрь `b.Path`. Нет произвольной записи по диску.
- Эндпоинт — admin-зона конфигуратора (как `ai-generate`/`ai-assist`).
- Содержимое — это YAML-метаданные (без исполнения); `.os`-модули в 2a/2b не генерируются.
- БД-базы исключены явно (запись в configdb — отдельный шаг).

## YAGNI / вне 2b

- **БД-базы (configdb) при применении** — отдельный шаг (валидация уже есть, добавить ветку записи).
- **Выборочное применение** (галочки по объектам) — пока «применить всё»; чекбоксы — позже при спросе.
- **Hot-reload рабочей базы** после применения — существующий механизм, не трогаем.
- Генерация `.os`-модулей и их применение через песочницу (этап 0) — после 2b.
